# Requirements Document

## Introduction

Sticks M3 delivers the tournament engine — the primary technical deliverable of the entire platform. M1 shipped auth, onboarding, the data model, GPS scoring, a basic leaderboard, and a social feed. M2 hardened background location, offline sync, course geometry overlays, and tee time booking via foreUP/GolfNow. M3 builds the white-label tournament and league management system that lets any golf organization — amateur tours, high school programs, golf academies, private clubs — run their competitive season through Sticks. The scope covers tournament creation and configuration (stroke play, match play, Stableford, scramble, best ball, Chapman), automated bracket and pairing generation seeded by handicap with bye handling, a commissioner dashboard for event management, real-time leaderboard updates via Supabase Realtime, white-label org branding (name, logo, colors), season management with cross-event standings, coach/admin permission levels, scoring exports (CSV/PDF), and a public spectator leaderboard URL that works without the app. The existing Organization, Tournament, TournamentEntry, Season, Round, and Hole models in the Prisma schema are extended — not replaced — to support these features.

## Glossary

- **App**: The Sticks mobile application built with Expo SDK 54 / React Native
- **API_Server**: The Bun + Hono backend server hosted on Railway, using Prisma ORM against Supabase Postgres
- **Commissioner**: A User with the COMMISSIONER role on an Organization who can create tournaments, manage registrations, generate pairings, post announcements, finalize results, and handle disputes
- **Coach**: A User with the COACH role on an Organization who has read access to rosters, standings, and player stats but cannot compete in tournaments
- **Admin**: A User with the ADMIN role on an Organization who has full management access equivalent to Commissioner but is designated as non-competing staff
- **Player**: A User registered as a TournamentEntry in a Tournament who competes and submits scores
- **Organization**: An entity representing a tour, league, school program, academy, or club that hosts tournaments and seasons through the white-label engine
- **Org_Config**: The white-label branding configuration on an Organization including custom name, logo URL, and color scheme applied to player-facing screens
- **Tournament**: An event hosted by an Organization with a defined format, handicap mode, flight configuration, and one or more rounds
- **Tournament_Format**: The scoring format for a Tournament — one of STROKE_PLAY, MATCH_PLAY, STABLEFORD, SCRAMBLE, BEST_BALL, or CHAPMAN
- **TournamentEntry**: A record linking a Player to a Tournament with flight assignment, tee assignment, seed position, and scoring data
- **Season**: A collection of Tournaments under an Organization with a points system, schedule, and cumulative standings
- **Season_Standings**: The ranked list of Players across all events in a Season, computed from the configured points system with optional drop-worst-round rules
- **Flight**: A grouping of Players within a Tournament by handicap range (e.g., A/B/C flights)
- **Bracket**: A single-elimination or double-elimination match play tournament structure generated from seeded Players
- **Pairing**: A grouping of Players assigned to play together in a round, generated by handicap seeding, random draw, or Commissioner assignment
- **Bye**: A placeholder assigned to a Player in a Bracket when the player count is not a power of 2, granting automatic advancement
- **Seed**: A Player's position in the Bracket or Pairing order, determined by GHIN index, previous event results, or manual Commissioner assignment
- **Leaderboard**: A ranked display of Players in a Tournament or Season, updated in real time via Supabase Realtime
- **Spectator_Leaderboard**: A public web page displaying the Leaderboard for a Tournament accessible via URL without requiring the App
- **Commissioner_Dashboard**: The set of screens in the App where a Commissioner manages tournaments, registrations, pairings, announcements, results, and disputes
- **Announcement**: A push notification sent by a Commissioner to all registered Players in a Tournament
- **Score_Dispute**: A challenge raised against a Player's submitted score, reviewed and resolved by the Commissioner
- **Scoring_Export**: A generated CSV or PDF file containing tournament results with configurable field mapping
- **Roster**: The list of Players registered to an Organization or Season
- **Points_System**: The configuration defining how Tournament results translate to Season_Standings points
- **Drop_Worst_Rule**: A Season configuration allowing a specified number of a Player's lowest-scoring events to be excluded from Season_Standings
- **Handicap_Allowance**: A configurable percentage of a Player's handicap applied for net scoring in a Tournament
- **Cut_Rule**: A multi-round Tournament configuration that eliminates Players below a score threshold after a specified round
- **Re_Seed**: The process of updating Bracket or Pairing positions between rounds based on current standings
- **Tee_Assignment**: The tee box (e.g., championship, regular, forward) assigned to a Player or Flight for a Tournament
- **FeedEvent**: A database record representing a discrete platform action emitted to the social feed
- **Supabase_Realtime**: The Supabase Realtime subscription system used for live leaderboard and feed updates

## Requirements

### Requirement 1: Tournament Creation and Configuration

**User Story:** As a Commissioner, I want to create and configure tournaments with flexible format options, handicap modes, flights, and multi-round settings, so that I can run any type of golf event through Sticks.

#### Acceptance Criteria

1. WHEN a Commissioner taps "Create Tournament" in the Commissioner_Dashboard, THE App SHALL present a tournament creation form collecting: name, Tournament_Format, handicap mode (gross, net, or both), start date, end date, and optional prize pool amount
2. THE App SHALL support selection of Tournament_Format from: STROKE_PLAY, MATCH_PLAY, STABLEFORD, SCRAMBLE, BEST_BALL, and CHAPMAN
3. WHEN the Commissioner selects net or both handicap mode, THE App SHALL display a Handicap_Allowance percentage input defaulting to 100%
4. WHEN the Commissioner enables flights, THE App SHALL allow definition of Flight boundaries by handicap range (e.g., A Flight: 0–10, B Flight: 11–20, C Flight: 21+)
5. WHEN the Commissioner configures a multi-round Tournament, THE App SHALL collect the number of rounds, optional Cut_Rule threshold and cut round number, and whether scoring carries forward across rounds
6. THE App SHALL allow the Commissioner to assign Tee_Assignment options per Flight or per individual Player
7. WHEN the Commissioner selects MATCH_PLAY format, THE App SHALL display bracket type selection: single-elimination or double-elimination
8. WHEN the Commissioner saves the tournament configuration, THE API_Server SHALL create a Tournament record in the database with status DRAFT and all configuration fields persisted
9. THE App SHALL allow the Commissioner to edit a Tournament configuration while the Tournament status is DRAFT or REGISTRATION_OPEN
10. IF the Commissioner attempts to edit a Tournament that is IN_PROGRESS, THEN THE App SHALL restrict edits to non-structural fields (name, prize pool, end date) and display a warning that format and flight changes are locked
11. THE Tournament creation form SHALL use the same data model for both Sticks-hosted events and commissioner-run self-hosted tournaments, distinguished by a hosting type field on the Tournament record

### Requirement 2: Player Registration and Roster Management

**User Story:** As a Commissioner, I want to manage player registration for tournaments and maintain organization rosters, so that I know who is competing and can generate pairings from the registered list.

#### Acceptance Criteria

1. WHEN a Tournament status is REGISTRATION_OPEN, THE App SHALL display a registration button on the tournament detail screen for eligible Players
2. WHEN a Player taps the registration button, THE API_Server SHALL create a TournamentEntry record linking the Player to the Tournament with payment status PENDING
3. THE Commissioner_Dashboard SHALL display the list of registered Players for a Tournament with name, handicap, flight assignment, and payment status
4. WHEN a Commissioner views the registration list, THE Commissioner_Dashboard SHALL allow the Commissioner to manually add or remove Players from the Tournament
5. WHEN a Player registers for a Tournament with flights enabled, THE API_Server SHALL auto-assign the Player to the appropriate Flight based on the Player's GHIN index and the configured Flight handicap ranges
6. THE Commissioner_Dashboard SHALL allow the Commissioner to manually override a Player's Flight assignment
7. WHEN a Commissioner views an Organization's Roster, THE Commissioner_Dashboard SHALL display all Players who have participated in any Tournament or Season under that Organization
8. THE API_Server SHALL enforce that a Player cannot register for the same Tournament more than once by maintaining the unique constraint on TournamentEntry (tournamentId, userId)

### Requirement 3: Automated Bracket and Pairing Generation

**User Story:** As a Commissioner, I want the system to automatically generate brackets and pairings from the registered player list seeded by handicap, so that I do not have to manually arrange matchups and groupings.

#### Acceptance Criteria

1. WHEN a Commissioner taps "Generate Pairings" for a STROKE_PLAY Tournament, THE API_Server SHALL group registered Players into Pairings of 2–4 players ordered by Seed (GHIN index by default)
2. WHEN a Commissioner taps "Generate Bracket" for a MATCH_PLAY Tournament, THE API_Server SHALL generate a Bracket structure seeded by the Players' GHIN index with the lowest-index Player as the top seed
3. WHEN the player count in a MATCH_PLAY Bracket is not a power of 2, THE API_Server SHALL assign Byes to the highest-seeded Players so that those Players advance automatically in the first round
4. THE API_Server SHALL support single-elimination and double-elimination Bracket types as configured on the Tournament
5. WHEN a Commissioner taps "Generate Groups" for a round-robin Tournament, THE API_Server SHALL divide Players into balanced groups with automatic advancement rules based on group standings
6. THE Commissioner_Dashboard SHALL allow the Commissioner to choose the seeding method: by GHIN index, by previous event results, or manual ordering
7. WHEN the Commissioner selects manual ordering, THE Commissioner_Dashboard SHALL display a drag-and-drop interface for arranging Player seed positions
8. AFTER generating Pairings or a Bracket, THE Commissioner_Dashboard SHALL display the result for Commissioner review before the Commissioner confirms and publishes the draw
9. THE Commissioner_Dashboard SHALL allow the Commissioner to manually swap individual Players between Pairings or Bracket positions after generation but before publishing
10. WHEN a Tournament has multiple rounds and Re_Seed is enabled, THE API_Server SHALL regenerate Pairings or Bracket positions between rounds based on current standings
11. THE API_Server SHALL persist all generated Pairings and Bracket data as structured records linked to the Tournament and round number

### Requirement 4: Commissioner Dashboard

**User Story:** As a Commissioner, I want a centralized dashboard to manage all aspects of my tournaments — registrations, pairings, announcements, results, and disputes — so that I can run events efficiently from my phone.

#### Acceptance Criteria

1. WHEN a User with the COMMISSIONER role opens the Commissioner_Dashboard, THE App SHALL display a list of the Commissioner's Tournaments grouped by status (DRAFT, REGISTRATION_OPEN, IN_PROGRESS, COMPLETED)
2. THE Commissioner_Dashboard SHALL provide navigation to: tournament configuration, registration management, pairing/bracket generation, announcements, results finalization, and score dispute resolution for each Tournament
3. WHEN a Commissioner taps "Post Announcement" for a Tournament, THE App SHALL present a text input and send the Announcement as a push notification to all registered Players via the push notification service
4. WHEN a Commissioner taps "Finalize Results" for a completed Tournament, THE API_Server SHALL lock all scores, compute final standings, and update the Tournament status to COMPLETED
5. WHEN results are finalized, THE API_Server SHALL emit a FeedEvent of type TOURNAMENT_RESULT containing the Tournament name, top finishers, and final standings
6. WHEN a Score_Dispute is submitted by a Player, THE Commissioner_Dashboard SHALL display the dispute with the Player's name, hole number, claimed score, recorded score, and optional evidence
7. WHEN a Commissioner resolves a Score_Dispute, THE API_Server SHALL update the Player's score record with the Commissioner's decision and notify the Player of the resolution
8. THE Commissioner_Dashboard SHALL allow the Commissioner to manually edit any Player's hole-by-hole scores for a Tournament round
9. WHEN a Commissioner manually edits a score, THE API_Server SHALL log the edit with the Commissioner's identity, timestamp, original value, and new value for audit purposes
10. THE Commissioner_Dashboard SHALL display a real-time view of scoring progress during an active round showing which Players have submitted scores for each hole

### Requirement 5: Live Tournament Leaderboard

**User Story:** As a Player or spectator, I want to see a live leaderboard that updates in real time during active tournament rounds, so that I can follow the competition as it happens.

#### Acceptance Criteria

1. WHEN a Tournament is IN_PROGRESS, THE Leaderboard SHALL display all Players ranked by their current score relative to par, updated in real time via Supabase_Realtime
2. THE Leaderboard SHALL display each entry with: rank position, Player name, avatar, score relative to par, total score, thru (holes completed), and score movement indicator (up/down/unchanged)
3. WHEN a Tournament has flights enabled, THE Leaderboard SHALL support filtering by Flight so that Players can view standings within their Flight
4. THE Leaderboard SHALL support a net/gross toggle when the Tournament handicap mode is set to "both"
5. THE Leaderboard SHALL display the top 3 Players with a podium treatment using gold, silver, and bronze visual indicators
6. THE Leaderboard SHALL display a "My Position" sticky card for the current Player that remains visible while scrolling a large field
7. WHEN a Commissioner freezes the Leaderboard, THE Leaderboard SHALL stop displaying real-time updates to Players and spectators and show a "Leaderboard paused" indicator
8. WHEN a Commissioner unfreezes the Leaderboard, THE Leaderboard SHALL resume real-time updates and reconcile all score changes that occurred during the freeze
9. WHEN a Player submits a score for a hole during an active Tournament round, THE API_Server SHALL broadcast the updated leaderboard position to all connected Supabase_Realtime subscribers within 2 seconds
10. THE Leaderboard SHALL support display for MATCH_PLAY Tournaments showing bracket progression, current match status, and hole-by-hole match results

### Requirement 6: Public Spectator Leaderboard

**User Story:** As a spectator without the Sticks app, I want to view a live tournament leaderboard via a shareable URL in my browser, so that I can follow the event from any device.

#### Acceptance Criteria

1. THE API_Server SHALL expose a public web endpoint that renders the Spectator_Leaderboard for a given Tournament ID without requiring authentication
2. THE Spectator_Leaderboard SHALL display the same ranking data as the in-app Leaderboard: rank, Player name, score relative to par, total score, thru, and movement indicator
3. THE Spectator_Leaderboard SHALL update in real time via Supabase_Realtime without requiring page refresh
4. THE Spectator_Leaderboard SHALL apply the Organization's Org_Config branding (name, logo, color scheme) to the page header and styling
5. THE Spectator_Leaderboard URL SHALL follow a shareable format (e.g., sticks.app/leaderboard/{tournamentId}) that can be shared via text, social media, or email
6. THE Spectator_Leaderboard SHALL be responsive and render correctly on mobile browsers, tablets, and desktop screens
7. WHEN a Commissioner has frozen the Leaderboard, THE Spectator_Leaderboard SHALL also display the "Leaderboard paused" indicator and stop updating
8. THE Spectator_Leaderboard SHALL display the Tournament name, Organization name, current round number, and event dates in a header section
9. IF a Tournament does not exist or has status DRAFT, THEN THE Spectator_Leaderboard SHALL display a "Leaderboard not available" message

### Requirement 7: White Label Organization Branding

**User Story:** As an Organization operator, I want to apply my custom name, logo, and color scheme to the player-facing tournament experience, so that my events look like they belong to my brand.

#### Acceptance Criteria

1. THE Commissioner_Dashboard SHALL provide an Organization settings screen where the Commissioner can upload a logo, set a custom Organization name, and configure a color scheme (primary color, secondary color, accent color)
2. WHEN a Player views a Tournament belonging to a branded Organization, THE App SHALL apply the Organization's Org_Config color scheme to the tournament detail screen, registration screen, and Leaderboard
3. WHEN a Player views the Spectator_Leaderboard for a branded Organization's Tournament, THE Spectator_Leaderboard SHALL display the Organization's logo in the header and apply the Organization's color scheme
4. THE API_Server SHALL store the Org_Config as a JSON field on the Organization record containing logo URL, primary color hex, secondary color hex, and accent color hex
5. WHEN an Organization has no custom Org_Config set, THE App SHALL fall back to the default Sticks branding (dark theme with primary #84d7af)
6. THE App SHALL validate uploaded logo images to ensure the file size is under 2 MB and the format is PNG or JPEG
7. THE Spectator_Leaderboard shareable link SHALL include the Organization branding so that the shared URL renders with the Organization's visual identity

### Requirement 8: Season Management and Standings

**User Story:** As a Commissioner, I want to create seasons that track standings across multiple tournaments with a configurable points system, so that I can run a full competitive season through Sticks.

#### Acceptance Criteria

1. WHEN a Commissioner creates a Season in the Commissioner_Dashboard, THE App SHALL collect: season name, start date, end date, associated Organization, and points system configuration
2. THE Points_System configuration SHALL allow the Commissioner to define point values for each finishing position (e.g., 1st = 100 pts, 2nd = 90 pts, 3rd = 80 pts) or use a predefined template
3. WHEN a Commissioner creates a Tournament, THE App SHALL allow the Commissioner to associate the Tournament with an existing Season
4. WHEN a Tournament associated with a Season is finalized, THE API_Server SHALL compute and update the Season_Standings by applying the Points_System to the Tournament results
5. THE Season_Standings SHALL display Players ranked by cumulative points across all finalized Tournaments in the Season
6. WHEN a Season has a Drop_Worst_Rule configured, THE API_Server SHALL exclude the specified number of each Player's lowest-scoring events from the Season_Standings calculation
7. THE Commissioner_Dashboard SHALL display the Season_Standings with Player name, total points, events played, events counted, and rank position
8. THE Commissioner_Dashboard SHALL allow the Commissioner to manage the Season Roster by adding or removing Players from the Season
9. THE Season detail screen SHALL display the schedule of Tournaments in the Season with dates, status, and links to each Tournament's detail screen
10. WHEN a Season's final Tournament is completed, THE Commissioner_Dashboard SHALL allow the Commissioner to finalize the Season, locking the Season_Standings

### Requirement 9: Coach and Admin Permission Levels

**User Story:** As an Organization operator, I want coach and admin roles that are separate from competing players, so that staff can manage events and view data without appearing on leaderboards.

#### Acceptance Criteria

1. THE API_Server SHALL support the following Organization-level roles: COMMISSIONER, COACH, ADMIN, and PLAYER
2. WHEN a User is assigned the COACH role on an Organization, THE App SHALL grant read access to: Rosters, Season_Standings, Tournament results, and individual Player scoring data for that Organization
3. WHEN a User is assigned the COACH role, THE App SHALL exclude that User from Tournament registration and Leaderboard rankings for that Organization's events
4. WHEN a User is assigned the ADMIN role on an Organization, THE App SHALL grant the same management permissions as COMMISSIONER (tournament creation, registration management, pairing generation, result finalization) while excluding the User from competing
5. THE Commissioner_Dashboard SHALL provide an interface for the Commissioner to invite Users to the Organization and assign them a role (COMMISSIONER, COACH, ADMIN, or PLAYER)
6. WHEN a Commissioner invites a User, THE API_Server SHALL create an Organization membership record linking the User to the Organization with the assigned role
7. THE API_Server SHALL enforce role-based access control on all tournament management endpoints so that only Users with COMMISSIONER or ADMIN roles can modify Tournament configuration, generate pairings, or finalize results
8. WHEN a Coach views a Player's scoring data, THE App SHALL display the Player's hole-by-hole scores, round history, and performance trends within the Organization's Tournaments

### Requirement 10: Scoring Export

**User Story:** As a Commissioner, I want to export tournament results as CSV and PDF files with configurable field mapping, so that I can submit scores to external bodies and share results outside the app.

#### Acceptance Criteria

1. WHEN a Commissioner taps "Export Results" for a finalized Tournament, THE Commissioner_Dashboard SHALL present export format options: CSV and PDF
2. WHEN the Commissioner selects CSV export, THE API_Server SHALL generate a CSV file containing: Player name, handicap, flight, gross score, net score (if applicable), score by hole, and final rank
3. WHEN the Commissioner selects PDF export, THE API_Server SHALL generate a PDF scorecard document containing the Tournament name, date, Organization name, and a formatted table of all Player results
4. THE API_Server SHALL support configurable field mapping for CSV exports so that the Commissioner can select which columns to include and define column header names
5. THE Scoring_Export SHALL include the Organization's branding (name and logo) in the PDF header
6. WHEN an Organization is configured for state association reporting (high school programs), THE API_Server SHALL support a state association export template that formats results according to the configured state's requirements
7. THE API_Server SHALL generate the export file and return a download URL that remains valid for 24 hours
8. THE Commissioner_Dashboard SHALL allow the Commissioner to share the export download URL via the device's native share sheet

### Requirement 11: Database Schema Extensions for M3

**User Story:** As a developer, I want the Prisma schema extended with tournament configuration, bracket/pairing structures, organization roles, season standings, and white-label branding fields, so that all M3 features have the data model they need.

#### Acceptance Criteria

1. THE Tournament table SHALL be extended with columns for: hosting type (STICKS_HOSTED, SELF_HOSTED), bracket type (SINGLE_ELIMINATION, DOUBLE_ELIMINATION, ROUND_ROBIN, or null), handicap allowance percentage, tee assignment configuration (JSON), multi-round configuration (number of rounds, cut rule, carry-forward flag), and leaderboard frozen flag
2. THE TournamentEntry table SHALL be extended with columns for: seed position, net score, points earned, match result (for match play), and elimination status
3. THE API_Server SHALL include a new Prisma model for Pairing with fields for: Tournament reference, round number, group number, and an ordered list of TournamentEntry references
4. THE API_Server SHALL include a new Prisma model for BracketMatch with fields for: Tournament reference, round number, match number, Player 1 TournamentEntry reference, Player 2 TournamentEntry reference, winner TournamentEntry reference, is_bye flag, and match score
5. THE API_Server SHALL include a new Prisma model for OrgMembership with fields for: Organization reference, User reference, role enum (COMMISSIONER, COACH, ADMIN, PLAYER), and timestamps
6. THE Organization table SHALL be extended to use the existing colorScheme JSON field for full Org_Config branding (primary color, secondary color, accent color) and ensure logoUrl is populated for white-label rendering
7. THE Season table SHALL be extended with columns for: roster (relation to OrgMembership or User list), finalized flag, and season standings cache (JSON)
8. THE API_Server SHALL include a new Prisma model for ScoreAuditLog with fields for: Tournament reference, TournamentEntry reference, hole number, original value, new value, edited by User reference, and timestamp
9. THE API_Server SHALL include a new Prisma model for ScoreDispute with fields for: Tournament reference, TournamentEntry reference, hole number, claimed score, recorded score, evidence URL, resolution status, resolved by User reference, and resolution notes
10. THE schema changes SHALL be managed through Prisma migrations so that they are version-controlled and reproducible
11. THE schema extensions SHALL preserve all existing M1/M2 data and relationships without breaking changes to the current Tournament, TournamentEntry, Season, and Organization models
